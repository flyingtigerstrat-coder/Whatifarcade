## 1. What we're building

**IRONLINE** — a mobile-first, pixel-art idle/management game about an upgradeable armored **battle train crossing a post-apocalyptic "rail ocean" wasteland.** Inspired by *Tiny Rails* / *Tiny Tower* (cozy, side-on, watch-your-train-roll) crossed with a Mad-Max wasteland (combat, raiders, scavenging, rust).

The core fantasy: *you build a rig, send it across the wastes, watch it fight and roll through a living world, pull into named depots, and grow it over a long journey.* It is a **feeling-first dream game** — whimsical, nostalgic, and beautiful — not a metrics-optimized product. See §6 for the collaboration mentality, which matters as much as the tech.

## 3. Environment & workflow constraints

- **No network / no npm.** Cannot install packages (node-canvas etc.). Validate JS by extracting the `<script>` block and `new Function(src)` parse-checking it in node.
- **Runtime testing harness:** because we can't render headlessly with real canvas, I built reusable node harnesses in `/tmp/` that **mock the DOM + canvas** and run the actual game loop / `drawTrain()` to catch crashes and (with an affine-matrix tracker) report device-space draw coordinates and color histograms. This is how I prove "no crash across every car type/weapon/tier/mode" and how I found bugs. Rebuild these as needed — they are the closest thing to a test suite. Pattern: stub `document.getElementById`→fake element, `getContext`→proxy ctx that records `fillRect` (optionally tracking `save/restore/setTransform/translate/scale/rotate` as a matrix), stub `requestAnimationFrame` to run N frames then stop, `eval(script)`, then optionally expose internals via appended `globalThis.__x=x;`.
  - **ASCII pixel-viewer (the key visual-debug tool):** a variant that records each `fillRect` color into a logical `grid[H][W]`, then prints a region as ASCII with a color→char legend — so I can *see* a car's silhouette/composition (or the rail-under-wheel alignment) before presenting, instead of guessing. Set `S.slots`/`S.ex` and call `drawTrain()` (or full `draw()`). To read solid geometry through the translucent overlays (`drawOverlays`/weather/vignette draw full-canvas gradients/rgba on top), **only record opaque `#`-hex fills** (skip gradient objects + rgba). It's a proxy (no real color/AA) — the partner's screenshot stays the final word, but it kills most round-trips.
  - **Durable PNG exporter (`ironline-export.js`, lives next to the game):** the productionised version of the above — `node ironline-export.js <subject> [opts]` renders any sprite straight from the game's draw code to a real PNG (true 1:1 snapshot, not a redraw). Subjects: `boss`/`boss-engine`/`boss-car`, `player`/`player-engine`, `rig <consist>` (compose any train, e.g. `rig gun:rocket,troop`), `raider`, `trader`, `raider-train`, `elite`, `all`. Opts: `--transparent --scale=N --tier=N --bg=#hex --out=PATH`. Same mock-canvas + matrix-stack + hand-rolled PNG-encoder infra as the harnesses, made permanent and parameterized. **Running it is sub-second/free; the only cost was ever re-authoring it**, which is now paid once and committed. Two facts baked in: the **matrix transform stack is mandatory** (player gun turrets `save/translate/rotate` — without it they render at the origin), and **single-slot `drawTrain` draws a phantom left-neighbour stub** so single cars are exported via `rig` (full-train path) with the left edge clipped. This is the seed of the offline asset pipeline (see §6); it belongs in the Claude Code repo as a standing command and is the natural host for the future PNG→indexed-array converter.
  - **Asset → sprite importer (`ironline-import.py`) + in-game blit path — the OTHER half of the pipeline.** `python3 ironline-import.py <asset.png> <name> [--width --palette=ironline|adaptive --dither]` takes a high-fidelity reference (hand-made or AI-generated) and produces a faithful *game-resolution* sprite: crop subject off the style-guide card by **luminance+saturation** (NOT modal color — the cards are a cream card in a white margin with a grid, which fools single-color bg tests), Lanczos downsample to the game grid, lock to the **curated IRONLINE palette** (built from the game's own ramps so imported + procedural share one color world), transparent bg, → tiny base64 PNG + an embed snippet `loadSprite(name,w,h,dataURI)`. The game side is `SPRITES`/`loadSprite`/`blitSprite(name,x,y,scale)` near the canvas setup (`ctx.imageSmoothingEnabled=false`; `y`=baseline the sprite bottom rests on; uses `ctx.drawImage` in 320×180 logical space). Safe in the exporter (mock `Image` never fires `onload` → `blitSprite` no-ops). **Architecture this sets up: HYBRID.** Hero/story trains (Fortress, named bosses, the player's signature engine) import as bitmaps for max beauty; the systemic fleet stays procedural (tiers/weather/variety stay free); animation (wheels/smoke/turret) layers over static hero bitmaps as procedural overlays. First hero sprite embedded: `fortress` (140×66, from the style-guide asset). Honest gap measured: style-guide assets are ~3–4× our linear detail + ~150 colors + painterly AA — don't chase that in `fillRect`, import it. Tuning TODO: trim the baked rail line from imported sprites; add `drawImage` to the exporter so heroes render in-engine headless.
  - **Exporter is now a full SCENE + ANIMATION renderer (`ironline-export.js`), not just sprites.** Three capability layers added on top of the sprite exporter, each a permanent, reusable tool: **(1) Scene rendering** — the mock canvas gained **real gradients** (`createLinearGradient`/`createRadialGradient` record stops; `fillRect` samples them per-pixel via `sampleGrad`) and a **path system** (`beginPath`/`moveTo`/`lineTo`/`arc`/`ellipse`/`fill`/`stroke` → scanline polygon fill + segment stroke), so the backdrop's sky/ground/haze/sun gradients **and** its `tri` mesas / `blob` debris / `ribArc` bones / ground cracks all render headless (they were silently no-ops before). `drawBackdrop` is exposed in `__G`, and a **`scene` subject** (`node ironline-export.js scene --off=N --T=N`) draws `drawBackdrop()`+`drawTrain()` as a full 320×180 world frame (via the new `fixedBox` writer mode). `--off` picks the world-position → biome, `--T` picks the time → day/night. This is what lets me iterate on the world headlessly instead of round-tripping screenshots. **(2) Animation** — `--anim=N` renders N **aligned** frames stepping `S.off` through exactly one driver rotation and `S.T` through one smoke cycle (seamless loop), union-bboxing them to a shared `fixedBox` so they line up; PIL then assembles a GIF + phase strip. `--off=N`/`--T=N` also step a single frame for stills. This is the reusable rig for *every* hero animation. **(3) The animation pattern** — `heroEngineFX(EX,eb)` in `drawTrain` layers procedural motion (cranking side-rod + orbiting crank-pin glints on the drivers, synced to `S.off`; a self-contained smoke plume drifting on `S.T`) over the static imported Fortress — the global smoke emitter is suppressed when `S.engineSkin` is set to avoid a double-plume. Net: imported hero + ~30 lines of overlay = a living engine. Gotcha baked in: gradient/path fills assume a **translate-only matrix** (true for the backdrop, `M`=identity) — fine for scenes, don't rely on it under rotation/scale.
  - **Hero sprite AS a live engine (`S.engineSkin`).** `drawTrain` checks `S.engineSkin`: if set + ready, it blits the hero at the engine slot (`blitSprite(skin,EX-1,BASE,S.engineSkinScale||1,flip)`) and gates out the procedural engine block + crane jib (the jib reuses `crX`/`mTop`, hoisted to function scope for a clean gate). **The Fortress asset already faces RIGHT (cowcatcher front-right), matching the player engine, so it blits with `flip=false`.** (`blitSprite`'s `flip` arg — save · translate · scale(-1,1) — stays available for any future asset that faces the wrong way; an earlier pass wrongly flipped the Fortress and made it face left — fixed.) Default (`engineSkin` null) is unchanged. **It's a live in-game upgrade now:** under the Engine card in the Train bay there's a **⛓ Fortress refit · `FORT_COST`(=500) scrap** button — a one-time unlock that sets `S.fortressUnlocked=true` + `S.engineSkin='fortress'`; once owned the card shows an **Engage / Revert** toggle (`fortressOn`/`fortressOff`, free) so the player isn't locked into the skin. Both `engineSkin` + `fortressUnlocked` persist via `save()`/`load()` (legacy-safe defaults `||null` / `!!`). It is a **visual hull skin** over the engine slot — `S.engine` level / hull / run-time are independent (could be tied to stats later). Graceful fallback: skin set but sprite not yet loaded → `ENGSKIN` is false that frame → the procedural engine draws until the bitmap is ready (never a blank engine). The Fortress sprite is **100×50** (re-imported down from 125×63 to fit the engine footprint — at 125px it overflowed the right screen edge in combat, where the engine parks at x≈216 in the 320-wide logical space). `blitSprite` takes a `flip` arg. Exporter: `--engine=NAME` sets the skin on a `rig`/`player` and widens the clip to `S.ex+145`; the mock `Image` decodes PNGs (`decodePNG`: inflate + unfilter, color types 6/2/0/3) and `ctx.drawImage` inverse-maps through `M` (honors translate/scale/mirror). So `node ironline-export.js rig oil,troop,gun:cannon --engine=fortress` renders the hero-headed train 1:1, headless. Next polish: procedural animation overlays (wheels/smoke) on the static hero. Parked: **Project Rubik's Cube** = modular paper-doll part-slots (chassis/cab/boiler/stack/turret/pilot/gear) from a parts library + indexed-palette recolor (`IRONLINE_engine_slotmap.png`).
- **Edit discipline:** all edits via `str_replace` on the one file. After each change: parse-check, then run the harness across configs. Present via `present_files`.
- **Persistence:** the game saves to `window.storage` (artifact persistent K/V) under key `battle-train-save`. Outside the Claude artifact host, saving shows "Saving unavailable here" — that's expected, not a bug.

## 4. Technical architecture (accurate to current code)

Single file, three parts: `<style>` (app-shell CSS), `<body>` (shell DOM), `<script>` (everything).

### 4.1 Rendering core
- **Canvas backing store is 640×360** (`<canvas width=640 height=360>`). A render-scale `const RS=2` is applied via `ctx.setTransform(RS,0,0,RS,0,0)` at the **top of every `draw()`**, so **all draw code is authored in a 320×180 logical coordinate space** and rasterizes at 2× density. This is the "sharpness" lever. **Never change logical coords to match the backing store** — keep authoring at 320×180.
- **Pointer hit-tests** map client→logical via `rect.width*320` / `rect.height*180` (NOT the backing size). Unaffected by RS. Keep it that way.
- `px(x,y,w,h,c)` = `fillRect` with `Math.round` on all args (the pixel-art primitive). `box()` = 1px dark outline + fill. `pxa()` = same as px (used in backdrop). `grad()`, `blob()`, `tri()` = gradient/ellipse/triangle helpers for the backdrop.
- **CSS `image-rendering:pixelated`** + `background:#d9a05e` on the canvas. That sandy-yellow bg is what shows if nothing draws — i.e. **a flat-yellow canvas means draw() is crashing** (debug clue, see §7).

### 4.2 Game state
One central mutable object **`S`** holds everything: resources (`scrap,food,fuel`), progression (`journeys,engine,caboose,maxSlots`), the train (`slots[]`), combat/run state (`mode,jt,dur,waves,wi,hull,raiders,boss,crate,tracers,booms,smoke,flash,turr`), world odometer (`off`), travel (`dist,depot,discount`), UI (`view,logHist`), passing train (`ptrain,...`), timing (`T,dt`), camera (`ex,pan,maxPan,stackX,panHintT`).
- **`slots[]`** entries: `null` (empty) or `{type:'gun'|'farm'|'troop'|'repair'|'oil', wpn?, port?, lvl, crew:{kind,rank}, hero?:{kind,rank}}`. `crew` is the role crew (lazy-created by `crewOf(s)`); `hero` is an optional assigned hero that overrides it (see §4.2b). Older saves upgrade transparently.
- **Save schema** (only these persist): `scrap,food,fuel,heroes,journeys,engine,caboose,maxSlots,slots,hull,dist,discount,layout`. **Crew and assigned heroes live *inside* each slot**, so they persist for free with `slots`; the separate `heroes` field is the **unassigned hero bench**. `S.crewSel` (bench selection for the assign UI) is transient — not saved, reset to −1 on load. `load()` has backward-compat (legacy guns→cannon, `fuel`/`heroes` defaulted, `crewOf` backfills crew on every slot). When adding persistent state, extend save+load+reset together.

### 4.2b Crew & heroes (per-car population + train-wide buffs)
**One crew member rides the back corner of each car** and buffs the *whole* train; recruited **heroes** can be assigned over a car's role crew.
- **`CREW`** catalog: role crew (`captain`/`sergeant`/`farmhand`/`mechanic`/`stoker`, each tied to a car `type`) + two **hero rosters** (each entry `hero:true` with a `grp`): **wanderers** `cook`/`medic`/`greaser`/`tinker` (`grp:'wander'`) and **spoils-of-war** `warboss`(`rare:true`)/`bombardier`/`ironside`/`reaver` (`grp:'war'`). `WANDER_HEROES`/`WAR_HEROES` are the recruit pools; `ROLECREW` maps car type→default crew kind; `RNK=['','I','II','III']`, `MAXRANK=3`. (`scavenger` kept as a legacy wanderer for save-compat.)
- **`crewOf(s)`** returns/lazy-inits the role `s.crew={kind,rank}`. **`activeCrew(s)=s.hero||crewOf(s)`** is *the* accessor everything reads — an assigned hero overrides the role crew for both buffs and drawing. **`crewBuffs()`** scans slots via `activeCrew` and aggregates `{gunMult,foodMult,fuelMult,scrapMult,hullRegen,hullDmgMult,hullFloor}` (Captain +10%/rk gun, Sergeant −5%/rk dmg, Farmhand +12%/rk food, Mechanic +1/rk regen, Stoker −6%/rk fuel; Warboss +10%/rk gun & +8%/rk scrap, Bombardier +9%/rk gun, Ironside −6%/rk dmg, Reaver +15%/rk scrap, Cook +18%/rk food, Greaser −7%/rk fuel, Tinker +1/rk regen, Medic hull-floor). Read live by `gun()`, `farm()`, regen/`hurt()`, reward payouts, `fuelC()`.
- **Heroes (bench + assignment):** unassigned heroes live on `S.heroes`. `recruitHero(group)` rolls `pickHero(group)` (the Warboss is a separate ~14% rare roll inside `'war'`), and either **ranks an owned copy** (`ownedHero`) or pushes a new one to the bench. Sources: `bossKill` → `recruitHero('war')` at ~85% (else scrap); the depot **recruit** offer and ~22% of passing **traders** → `recruitHero('wander')`. Assignment is a two-tap Crew-tab flow (`S.crewSel` selects a bench hero, then tapping a car runs `heroPlace`; `heroReturn` sends it back; swapping is handled). Training (`crewRankCost` scrap) ranks whatever is **active** in the slot (hero if assigned, else role crew).
- **Drawing:** `crewFig(x,kind,rank)` is the shared standing-figure (leg gap, kind-distinct headgear, gold rank pips). Every kind including all heroes has a headgear branch (war heroes red-accented, the Warboss horned + gold-crowned, wanderers gold-crested). The gun **Captain** keeps his cupola; a hero assigned to a gun car draws as a figure at the back corner. Cars read `activeCrew(s)`, so placing a hero reskins the figure automatically.

### 4.3 Economy / derived values (all pure functions of S)
`gun()` (×`crewBuffs().gunMult`), `troops()`, `farm()` (×`foodMult`), `repairRate()`, `oilProd()` (fuel/s from oil cars), `maxHull()=60+engine*30`, `loot()=1+caboose*0.1`, `base()` (run reward), upgrade costs `enC/cbC/slC/upC`. Difficulty scales off `S.journeys`.
- **Three-resource loop (rewired this session):** **scrap** = upgrades + builds + crew training; **fuel** (oil cars, `oilProd`) = the **movement gate**, `fuelC()` burned on departure; **food** (farms) = the **growth gate**, `carFoodC()` spent to weld a car, plus a small `rations()` crew nibble per departure. (Legacy `fdC()` — old food-on-depart — is dead, left harmless.) `NEWCAR=40` scrap to weld. `hurt(raw)` applies Sergeant mitigation + Medic hull-floor at every damage site.
- **Weapons are a two-slot modular system** (not a single power value — the legacy `WPOW` still exists but the live path is `wpow(top,port)`). Each gun slot has a **TOP turret** (`TOPW`: cannon/rockets/mortar) and a **broadside PORT** (`PORTW`: autocannon/flak/flamer). Each entry carries `pow`, a `dt` damage-type tag (`kinetic`/`blast`/`fire`, groundwork for future enemy weaknesses), and a tracer color `c`. `gun()` sums `lvl*wpow(wpn,port)` across gun slots (then ×`gunMult`). Weapon swaps cost `REFIT=30` (the `dc()`-discountable refit buttons on the car card).

### 4.4 Main loop
`tick(ts)` → compute `dt`, advance `S.off += spd*dt` (spd = 110 running / 14 idle — the world scrolls, train appears stationary), run simulation by `S.mode`, then `draw()`, then `upd()` (DOM/HUD), then `requestAnimationFrame(tick)`.

**`draw()` order (back→front, critical for depth layering):**
1. `drawBackdrop()` — sky/biome/weather/day-night/parallax (see §4.7)
2. `lane(122)` — far rail
3. smoke particles
4. `drawPtrain()` — passing train on the far rail (distant)
5. `lane(138)` — mid rail (**the train's own rail — moved up from 144 this session so the wheels sit *on* it**, not 6px above; shadow band `(16,139,294,4)` tucked under)
6. `drawTrain()` — **player train** (the hero)
7. `lane(164)` — near rail
8. idle tumbleweed
9. `S.raiders.forEach(drawRaider)` — foreground skirmishers
10. `if(S.boss)drawBoss()` — foreground boss
11. `drawTracers()` — projectiles
12. booms/flash
13. `drawOverlays()` (vignette/flash) → `drawPanHint()` (edge chevrons) → journey progress bar (`mode=='run'`)

So enemies draw **after** the train → they correctly occlude it as foreground objects. Preserve this order.

> **Camera note:** `drawTrain()` is the *only* thing anchored to the live camera value `EX = S.ex + S.pan` (see §4.9). Everything else — backdrop, lanes, raiders, boss, crate, smoke particles, tracers, the progress bar, and `drawPanHint` — is drawn in **fixed world/screen coordinates**. That separation is deliberate and load-bearing: combat entities sit at absolute coords and the train's turret-x is derived from the *same* `EX`, so aiming stays consistent at any camera position **as long as the camera holds native 216 while enemies are present.**

### 4.5 App-shell UI (DOM)
Persistent flexbox column `#app` (height `100dvh`): top stat bar (`#topbar` brand + **4 stockpiles: scrap / food / fuel / hull**) → canvas (`.frame`) → log ticker (`#ticker`) → swappable content (`#content`) → persistent action button (`#actionbar` #go) → bottom tab bar (`#tabs`).
- **Router:** `S.view ∈ {travel,train,crew,map,log}`; `showView(v)` toggles `.view` hidden + active tab and renders the entered view (`cards()` for train, `renderCrew()`, `renderMap()`, `renderLog()`) — so spending in one panel is reflected when you switch to another. Tab clicks + content driven by id.
- **Views:** `#v-travel` (next-stop hint + depot panel + status), `#v-train` (the `#cars` upgrade grid — pure rig-building + reset/savemsg), **`#v-crew`** (the `#crewlist` panel — crew summary banner + per-car crew cards; its own tab as of this session), `#v-map` (upcoming depots list — node-graph seed), `#v-log` (history).
- **km is *not* a stat-bar stockpile** — it's journey state, shown in the next-stop line (`#nextdepot`: `"<dist> km · ▸ next stop <name> — <rem> km"`, adapts to docked/crossing), visible in both orientations.
- `#go` is contextual: Depart (**`fuelC()` fuel** now, not food) / "Crossing…" / docked→"open depot". Tab `travel` gets `.alert` when docked; tab `crew` gets `.alert` when a crew is affordable to train.
- Scrollbar themed (rust/brass) via `::-webkit-scrollbar` + Firefox `scrollbar-color`.
- **`log(t)`** pushes to `S.logHist` (cap 16), updates `#ticker`, refreshes log view. Everything UI is by `id` — moving DOM is safe as long as ids are preserved.
- **Dual orientation.** The shell renders in two layouts driven by an `.land` class on `#app` (`S.layout`, persisted + auto-detected): **portrait** (the column above) and **landscape** (full-bleed canvas, corner stats, a consolidated top route header merging next-stop + biome, slim bottom tabs, and a filmstrip car tray). `applyLayout()` toggles it.
- **Display menu.** A single ⚙ button `#dm` toggles a popover `#dmpop` (container `#dispmenu`) holding three controls: **`#lt`** layout toggle, **`#fs`** fullscreen, **`#im`** immersive. It lives as a **direct child of `#app` at `z-index:50`** so it isn't trapped in the topbar's stacking context (that was a bug). Closes on select / outside-tap.
- **Immersive mode** (`S.immersive`, `applyImmersive()`, *not* persisted): strips all chrome, leaving only the world + the ⚙ to exit. Full-bleed in landscape, letterboxed in portrait.
- **Fullscreen** (`#fs`): `requestFullscreen` (+ webkit), synced via `fullscreenchange`. **Hard limit:** blocked inside the Claude artifact iframe (no `allow="fullscreen"` from the parent page — uncontrollable from content); `fsFail()` degrades to "↗ Open in tab". Only works on the downloaded file in its own browser tab.

### 4.6 Travel / depots
Odometer-based foundation (toward a future node-graph). `DEPOT_KM=20`. `maybeDepot()` (called from `finish/bossKill/bossEscape`-success) advances `S.dist += 4+floor(engine*0.6)` (faster engine = more range) and, on crossing a depot boundary, `openDepot(idx)`. Depots are **deterministic by index** via `hash()`: `depotName` (DPRE×DSUF word lists), `depotFlavor`, `depotOffers` (3 seeded from pool: repair/food/scrap/**fuel**/**recruit**/scavenge/cache/discount; always ≥1 free). The **fuel** offer is a hard-lock backstop (top off without an oil car); the **recruit** offer signs on a **wanderer** hero (`recruitHero('wander')`). Player picks **one action then leaves** — the replayable choice. `discount` buff halves the next upgrade (threaded via a `dc()` helper in the upgrade handler, cleared after one purchase). **North star: replace odometer with an FTL/Slay-the-Spire node-graph map** (branch choices slot exactly where the depot single-choice lives).

### 4.7 Living world (backdrop)
`drawBackdrop()` composites layers each frame: **biomes** (`BIOMES=[RUST,CITY,REEF]`, `BIOME_LEN=1150`, cross-faded at region seams, each with `.far/.mid/.deb/.land/.gtex/.wx`) → **weather** (5 moods W_CLEAR/DUST/ASH/FOG/BLOOD, biome-weighted, own odometer `WX_LEN`, `computeWeather`/`applyWeather`) → **day/night** (`dayNight()`, `DAY_LEN`, sun/moon arc, stars/milky-way/meteors at night). Procedural props are placed by `hash()` seeded on world position (`eachCell`, `cellBiomeIdx`), plus curated landmarks, region-name captions, and rare story props (`storyAt`). Order of intent: **time = light, weather = mood, biome = land.**

### 4.8 Combat
Depart costs **fuel** (`fuelC`, the movement gate) + a small food `rations()` → `mode='run'`. Non-boss runs spawn 3 raider waves (`wave()` resolves `gun()+troops*1.5` vs scaling enemy strength → hull damage or repel), optional tappable **salvage crate** (`grabCrate`, 65% good / 35% ambush), and optional **passing-train** encounter (`ptrain`: trader gift / raider firefight — a peaceful trader has a ~22% chance to bring a **wanderer hero** instead of supplies). Every 5th journey = **boss convoy** (`S.boss`, HP bar, `bossKill` / `bossEscape`), now staged as a **stop-and-duel**: when the convoy locks the line it sets `boss.set`, the **world scroll freezes** (`spd=0`) and the **crossing timer freezes** (`S.jt` held), there is **no countdown** — the crossing completes only on `bossKill`, and the bottom progress bar shows the **boss's HP depleting** as progress. The loss path is hull→0 → `bossEscape` (repurposed into the clean "limp home", no journey advance). Separately, a rare **elite passing raider** (`p.elite`, ~13% on any non-boss crossing past journey 3) is the *mirror*: a same-direction **broadside duel** on the parallel rail where the **world keeps rolling** but the crossing clock freezes while it's engaged (`state!='out'`); it never breaks off, resolves by kill (big scrap + 50% Warboss roll, then the crossing resumes) or hull-loss. **Crew shape combat:** all hull damage flows through **`hurt(raw)`** (Sergeant/Ironside `hullDmgMult` + Medic `hullFloor`); rewards scale by `scrapMult`; a boss kill has an ~85% chance to drop a **war hero** (`recruitHero('war')`, Warboss rare) else bonus scrap. Hull→0 = limp home with partial reward. **Gun turrets** (`S.turr[i]`) swivel to track nearest threat via `fireTarget()`/`nearRaider()`, recoil, and `spawnShot(kind,...)` into `S.tracers`. Gun cars also fire **broadside ports** (`spawnPort`) on timer `st.pt`. **This turret aiming math is load-bearing — preserve it verbatim when restyling gun cars; only reposition the pivot `pvy`.**
- **Combat coords are absolute** and *not* camera-relative: raiders spawn at `x:340+i*26`, boss at `x:380`→`tx:200`, crate at `x:340`. The turret's firing x is `pvx=cx+floor(carW/2)` — derived from the same `EX` the train draws at — so it stays aligned with those absolute targets **only while the camera sits at native 216** (which the camera guarantees whenever enemies are present; see §4.9).

### 4.9 Camera & input (the follow-camera + drag-to-pan)
The train can be longer/wider than the 320px stage, so a camera frames it. Two stacked offsets, both folded into a single anchor **`EX = S.ex + S.pan`** used only by `drawTrain`:
- **`S.ex` — auto follow-camera** (updated each `tick`): pins to **native 216 while enemies are present** (`S.boss || S.raiders.length`) so aiming/approach-distances stay calibrated; otherwise frames the consist with **open track ahead** — idle target `min(190, 140-(95-L)/2)` (cap lowered 216→190 this session so ~34–40px of rail shows in front of the engine's nose for a better front-facing/approach view), centering short consists and front-anchoring long ones (tail trails off-screen left). Lerped — leans forward into a fight (a gentle ~30px glide to 216), pulls back in peace.
- **`S.pan` — manual drag offset**, clamped `0..S.maxPan` where `maxPan = max(0,(L+10)-S.ex)` (full pan lands the caboose rear at x≈10). Drag **right** to reveal the tail (content-drag). Suppressed + lerped to 0 while enemies are present (combat re-frames the front; a wave can't be dodged by holding a drag). Short trains that already fit → `maxPan=0`, no pan.
- **`carW=40` fixed** (was `min(32,…/segs)` which shrank with count). Width is now decoupled from car count — the room for the per-type detail pass.
- **Input:** canvas pointer handlers split **tap vs drag** by a 3px threshold (pointer-captured). Tap = crate grab (hit-test unchanged, fires on `pointerup` when not dragging); drag = pan. So panning never eats a crate and vice-versa.
- **`drawPanHint()`** draws faint pulsing edge **chevrons** (left = tail to reveal, right = engine off-frame), brightened by `S.panHintT` while dragging, suppressed in combat.
- **Smoke follows the camera:** spawn x is `S.stackX` (stored by `drawTrain` each frame = current stack x), not a constant — otherwise it detaches when the anchor moves. Any *new* train-attached effect must do the same.

## 5. Art direction (the current focus) — READ BEFORE TOUCHING SPRITES

### 5.1 The reference & the target
The partner provided reference art (the "TINY RAILS — WASTELAND TRAINS" roster + a zoomed "War Rig" engine). The target visual language, decoded from the zoom:
1. **Two metals, not one:** bare **gunmetal grey** with **rust eating across it in irregular orange-brown patches** (grey on worn/raised areas, rust pooling in recesses). This is the #1 thing that reads as "real."
2. **~5 tonal values per form** + a hard black AO line where parts meet. Cylinders (boiler) get a horizontal specular highlight band; plates get a lit top lip + shadowed underside.
3. **Silhouette loaded with asymmetric scavenged junk** — welded spikes at *uneven* angles, hanging chains, skull clusters, tarps. The *randomness* sells it; tidy grid-aligned detail does not.
4. **Directional grime:** soot darkens **upward** near stacks; rust streaks **downward** under bolts/seams; dark pools in recesses.
5. **Per-faction palette identity:** War Rig = rust-orange; Raider = gunmetal-black; Fortress = cold grey; Nomad = warm brown; Ghost Hauler = bone-white + neon; Hunter = dark cage/bone.

Also note Tiny Rails proper (other references): a clean cozy parallax desert/night world, themed novelty trains (genie lamp, etc.), bottom toolbar UI, day/night. Our world already echoes this; the *wasteland* skin is our twist.

### 5.2 The honest resolution constraint
The reference engine is drawn across ~250–300px wide with per-pixel detail. **Our in-game engine lives in ~46×46 logical px (~92 device).** That's an ~8–10× pixel-budget gap. **We cannot reproduce the reference's full density at current size** — cramming it makes noise. So art work splits into:
- **Achievable now (the "material system"):** two-tone metal+rust, 5-value ramps, directional soot/rust, AO seams, a *few* asymmetric junk pieces. Same *language*, lower *resolution*.
- **Needs a resolution decision (partner's call):** bump `RS` 2→3 or raise the logical train band so the engine gets ~70–90px to work in, enabling more of the fanned spikes/skull clusters to actually read. A performance + full-retune decision — don't do it unilaterally.

> **UPDATE (2026-06-19) — the constraint now has a third answer: the asset → sprite import pipeline.** Rather than only choosing between (a) approximating the reference in `fillRect` or (b) a global resolution bump, **hero/story units can be imported directly** as high-fidelity bitmaps via `ironline-import.py` (downsample the *real* reference to the game grid, palette-lock to IRONLINE, blit via `blitSprite`), while the procedural systemic fleet stays untouched. This is the **HYBRID architecture** and it's the recommended path for the trains that carry the story — they no longer have to be hand-approximated over many passes. The **Fortress** is the first hero through it and is now a **live in-game engine upgrade** (see §3 tooling + the 2026-06-19 changelog entries). The resolution-bump decision (b) is therefore no longer the *only* way to get reference-grade heroes — it's now purely about the procedural fleet's density.

Plus three **weapon metals** (same 5-value shape, used by the reworked turrets): **`GUNM`** (cool forged steel), **`ROCKM`** (military green), **`BRASSM`** (mortar brass).

- **Helpers:** `panel(x,y,w,h,m)` (lit top lip + shadow underside + AO + side shadows), `cyl(...)` (cylinder w/ specular band), `streak(x,y,len,seed)` (downward rust drip), `rustPatch(x,y,seed)` (seeded rust fleck + drip), `spike(x,y,len,ang)` (welded spike at an angle), `skull(x,y)`, `chain(x,y,len)`.

## 7. Gotchas & hard-won lessons

- **Flat-yellow canvas = a crash in `draw()`** (canvas bg `#d9a05e` showing through). Don't assume it's the scale transform — historically it was a **stranded salvage crate**: crate movement/cleanup lived inside `if(mode=='run')`, so a crate present at crossing-end froze and drew forever. Fixed by clearing `S.crate` on every crossing-end path AND gating `drawCrate()` to `mode=='run'`. Lesson: **clear run-only state on every exit path; gate run-only draws by mode.**
- **Name collisions** with biome constants — `RUST` was taken (→ `RUSTR`). Grep before declaring a new top-level const.
- **`hash()` is a hoisted function declaration**, so helpers defined earlier can call it. Fine, but know it.
- **Inner `ctx.save/restore` must stay balanced** (turrets, tracers, biome land). The `setTransform(RS,...)` is re-applied fresh each frame at the top of `draw()`, so leftover transforms don't accumulate.
- **Preserve turret math verbatim** when restyling gun cars (§4.8).
- **Camera coupling (§4.9):** `EX = S.ex + S.pan` is dynamic. The camera *must* hold native **216 whenever enemies are present** — combat/crate/boss/raider coords are **absolute** and turret-x derives from `EX`, so they only align at 216. Don't make combat coords EX-relative. Any new **train-attached effect** must follow the anchor (spawn from `S.stackX` / an `EX`-relative x), or it detaches when the camera pans. When changing `carW` or engine length, re-check the `maxPan` math and the centering offset.
- **Extend save/load/reset together** when adding persistent state. **Crew is the exception that proves the rule:** it lives *inside* `slots`, so it persists for free — but `load()`/`reset()` must call `crewOf(s)` on each slot to backfill it on old saves.
- **Crew rank is data, not derived.** Each car's drawn rank pips read `activeCrew(s).rank` (real, trainable), *not* the car's level. Don't reintroduce a level→rank shortcut.
- **`activeCrew(s)=s.hero||crewOf(s)` is the one accessor** for buffs *and* drawing. An assigned hero overrides role crew. The unassigned **hero bench is `S.heroes`** (a separate saved field); `S.crewSel` is transient assign-UI state (not saved). When touching crew, go through `activeCrew`, not `crewOf`, or heroes silently stop applying.
- **Swiveling-weapon draw order (the rocket bug):** draw the mount/barbette/cradle **before** the rotated weapon, the weapon on top, and **never a world-space element after** it — otherwise it stamps over the weapon (the old grey housing occluded the compact rocket pod). Weapons use the `GUNM`/`ROCKM`/`BRASSM` ramps; the gun **roof is an integrated emplacement** (steel gun-deck + barbette + coaming, §5.4), not a flat plank.
- **Economy roles are rewired:** **fuel** gates departure, **food** gates growth (building cars) + rations. Don't reattach the departure cost to food (the dead `fdC()` is a trap). All hull damage must go through **`hurt()`** so Sergeant mitigation / Medic floor apply uniformly.
- **The train's rail is `lane(138)`** (mid lane), aligned to `BASE`/wheel contact. The three rails are now evenly spaced at **far `lane(110)` / mid `lane(138)` / near `lane(166)`** (28px gaps); the far rail sits **above** the train roof (`DECK=112`) so passing trains (drawn at `y=108`) ride clear of the player. If you move `BASE` or the wheel radius, move the mid lane with it (and the `(16,139,...)` shadow) or the rig floats again.
- **The boss is transient (`S.boss`), the elite passing raider lives in `S.ptrain`.** Neither the boss convoy nor the elite is in the save schema — they're regenerated on spawn. Don't try to persist them.
- **Two boss-class fights, deliberately opposite:** the head-on boss **stops the world** (`spd=0` + `S.jt` frozen, gated on `bossKill`); the elite passing raider **keeps the world rolling** (only `S.jt` frozen while `state!='out'`). When adding a new "fight until one wins" encounter, decide which staging it is and freeze the matching things. Both resolve by enemy-HP→0 (win) or hull→0 (loss); there is **no timer** on either.
- **Layout: `autoLayout()` vs `layoutManual`.** Wide windows auto-select the `land` (desktop) skin at boot and on resize *unless* `S.layoutManual` is set (the ⛶ toggle sets it). A non-manual stale `portrait` auto-corrects to desktop on load. The desktop canvas uses **`object-fit:contain`** to fill without distorting the 640×360 pixel art (dark frame on odd aspects) — don't revert it to `width/height:100%` (that stretches).
- **Headless harness needs a fuller `window` stub.** Top-level script now calls `window.addEventListener('resize',…)`, so the test mock's `window` must implement `addEventListener` (and `matchMedia`) or the whole script throws on load. Game code is fine in a real browser.
- **ASCII viewer reading through overlays:** `drawOverlays`/weather/vignette paint full-canvas gradients + rgba *on top*, so a naive last-write grid shows only the overlay. Record **only opaque `#`-hex fills** to read the solid scene (rails/wheels/sprites).
- **`showView` re-renders Train and Crew on entry** — both read live `S.scrap`, so spending in one tab stays consistent. Keep that if you add cross-panel currency UI.
- Validate every change: parse-check + harness across all car types / weapons / tiers / modes (idle, run, depot, boss) before presenting.
